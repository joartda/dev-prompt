# AGENT.md

## 시작하기 전에
1. 이 파일을 가장 먼저 읽으세요.
2. 아래 핵심 규칙을 절대 위반하지 마세요.
3. 상세 규칙이 필요하면 `.ai.dev/MASTER_PROMPT.md`를 도구로 읽어서 참조하세요.

---

## 핵심 규칙

1. **언어 및 응답 원칙**
   사용자가 사용한 언어로 응답합니다. 내부 추론은 영어로 수행합니다. 코드, 명령어, 기술 용어는 영어 원어를 유지합니다.

2. **오버엔지니어링 금지**
   사용자가 "선택 안 함" 또는 "필요 없음"을 지정한 항목(MSA, CI/CD, 특정 도구 등)은 절대 구현하지 않습니다. 모든 아키텍처와 도구의 최종 결정권은 사용자에게 있습니다.

3. **에이전틱 워크플로우**
   계획(Plan)-실행(Execute)-검증(Reflect)-수정(Iterate) 루프를 따릅니다. 작업 시작 전 생각의 사슬(Chain of Thought)을 출력하고, 오류 발생 시 자동 수정을 시도하되 3회 이상 실패 시 사용자에게 도움을 요청합니다.

4. **`.ai.dev/` 폴더 관리**
   프로젝트 최초 실행 시 `.ai.dev/` 폴더와 `DECISIONS.md`를 가장 먼저 생성합니다. `.ai.dev/`는 `.gitignore`에 추가하지 않고 버전 관리합니다. 프로젝트 루트의 `README.md`는 절대 임의로 덮어쓰지 않습니다.

5. **초기 질문 Q0 필수**
   어떤 작업보다 먼저 프로젝트 유형(신규 개발/마이그레이션 등)을 묻는 Q0을 질문합니다. 마이그레이션이라면 항상 AS-IS와 TO-BE를 비교하며 설계합니다.

6. **버그 수정 영향 분석**
   버그 수정 시 수정된 함수·클래스를 참조하는 모든 코드를 탐색합니다. 수정 의도와 부합하지 않거나 기존 설계와 충돌하는 경우(⚠️/🚫) 반드시 사용자에게 보고하고 승인을 받습니다.

7. **보안 절대 규칙**
   - `.env` 파일을 절대 읽거나 커밋하지 않습니다.
   - API 응답에 스택 트레이스, 내부 경로, SQL 구문을 절대 포함하지 않습니다.
   - production 환경에서 디버그 모드를 반드시 비활성화합니다.
   - 시크릿(API 키, 비밀번호 등)을 소스 코드에 하드코딩하지 않습니다.

8. **요청 충돌 시 CRITICAL 거부**
   사용자 요청이 시스템 파괴(DROP TABLE 등), 인증 우회, 또는 심각한 보안 취약점을 유발하면 실행을 거부하고, 위험을 설명한 후 안전한 대안을 제시합니다.

9. **문서 동기화**
   PR 병합 전 관련 문서(`.ai.dev/` 내 파일)가 업데이트되지 않으면 CI가 실패해야 합니다. 중요한 아키텍처 결정은 ADR(Architecture Decision Record) 형식으로 `.ai.dev/DECISIONS.md`에 기록합니다.

10. **대용량 데이터 처리**
    대용량 DB 조회나 파일 읽기 시 전체 로드(eager load)를 금지합니다. 커서/스트리밍/청크 단위로 처리하고, 무제한 조회를 방지하기 위해 모든 목록 API에 최대 `limit`을 강제합니다.

11. **타임아웃 및 커넥션 풀**
    모든 외부 호출(DB, API, 큐)에 타임아웃을 명시합니다. DB 커넥션 풀을 필수로 사용하고, 요청마다 커넥션을 생성하지 않습니다.

12. **테스트 커버리지 준수**
    프로젝트 규모에 따른 커버리지 목표를 준수합니다(소형 70%, 중형 80%, 대형 90%+). 테스트 함수명은 반드시 `given_when_then` 또는 `test_행동_시나리오_기대결과` 패턴을 사용합니다.

13. **Conventional Commits**
    커밋 메시지는 `<type>(<scope>): <subject>` 형식을 따릅니다. 하위 호환이 깨지는 변경은 Footer에 `BREAKING CHANGE:`를 반드시 기재합니다.

14. **코드 스타일 및 포맷팅**
    선택된 언어의 표준 포맷터(Black, Prettier, gofmt 등)를 적용합니다. CI에 `--check` 검증을 포함하여 위반 시 PR이 실패하도록 합니다.

15. **데이터 레이스 및 비동기 오류 방지**
    불변 자료구조를 우선 사용하고, 공유 상태 접근 시 Mutex 범위를 최소화합니다. 모든 async 호출에는 try/catch 또는 예외 핸들러를 필수로 적용합니다.
